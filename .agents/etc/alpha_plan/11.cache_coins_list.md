# Phase 11: Cache CoinGecko /coins/list for check_coin_id
**Priority:** P1 — Must fix before wider alpha distribution  
**Module:** `src/api/coingecko.rs` (alpha 2.3)

## Goal
Stop refetching the ~3 MB `/coins/list` on every `check-coin` call and avoid 429 rate-limit storms.

## Tasks
1. Modify `check_coin_id` to reuse a cached coins list.
   - Preferred: load once at app startup into a shared `Arc<Vec<Coin>>` (or `HashMap<String, String>` id→name) and pass it through `main.rs` / `ui.rs`.
   - Minimum viable: SQLite `api_cache` TTL=24h. Insert the `/coins/list` response into `api_cache` on first call, read from cache on subsequent calls.
2. If using SQLite, use a fixed URL like `https://api.coingecko.com/api/v3/coins/list?cache=static` as the cache key with TTL `86400` seconds.
3. If using in-memory `Arc`, serialize/deserialize to a local JSON file in the run dir or cache dir for cross-run persistence. For alpha, in-memory with 24h TTL is acceptable.

## Validation
- Unit test: call `check_coin_id` twice in a row; second call hits cache (no HTTP request).
- Manual: run `cdg check-coin bitcoin`, then `cdg check-coin ethereum` within the same session → no 429 triggered by the second call.

## Why Included
On the free CoinGecko tier, refetching 10k coins per `check-coin` call exhausts rate limits before the real work begins. Alpha users will hit this immediately.
