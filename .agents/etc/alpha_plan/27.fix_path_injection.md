# Phase 27: Wire path safety into coin-derived file paths
**Priority:** P1 / P2 — Security + correctness
**Module:** `src/pipeline.rs`, `src/main.rs`, `src/utils.rs`
**Source:** `code_review.md` #3

## Goal
Stop interpolating raw user-supplied coin IDs directly into filesystem paths. `sanitize_name` and `validate_safe_path` already exist in `utils.rs` but are never called before `std::fs::write` / `create_dir_all` in the fetch flows.

## Tasks
1. In `run_pipeline_flow` (and `run_ohlcv_flow`), before building any path from a coin id `c`:
    - `let c = utils::sanitize_name(c);` then `utils::validate_safe_path(&format!("{}/{}", base_dir, c))?;`
    - Apply to: run dir creation, OHLC json path (`{ohlcv_dir}/{c}_{curr}.json`), and any per-coin output file.
2. Same treatment in `main.rs` where coin-id interpolation builds `currency_dfs` / OHLC paths.
3. Add a unit/integration test: pass a malicious coin id like `"../../etc/passwd"` → `validate_safe_path` returns `Err` and no file is written outside the run dir.
4. Keep `sanitize_name`'s whitelist rewrite in `backlog.md` (deferred) — this phase only wires the existing guards.

## Validation
- `cargo test -p cdg` passes.
- Manual: `cdg collect --coin "../../etc/passwd"` → clean `Err`, no traversal, no file written outside run dir.

## Why Included
`code_review.md` #3 is a Warning-severity path-injection vector. Cheap to close by wiring existing utilities. Not alpha-blocking for trusted single-user CLI use, but required before wider alpha distribution.
