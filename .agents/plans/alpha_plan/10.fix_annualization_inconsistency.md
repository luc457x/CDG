# Phase 10: Fix annualization inconsistency in standalone backtest
**Priority:** P1 — Must fix before wider alpha distribution  
**Module:** `src/pipeline.rs`, `src/main.rs`  
**Source:** `base_plan.md` P4

## Goal
Make `run_standalone_backtest` respect the same annualization logic as `RunPipeline` (252 for weekdays, 365 for crypto/7-day) instead of hardcoding `365.0`.

## Tasks
1. Add `annualization_factor: Option<f64>` (or `drop_weekends: bool`) parameter to `run_standalone_backtest`.
2. Compute `let ann = annualization_factor.unwrap_or(if drop_weekends { 252.0 } else { 365.0 });` inside `run_standalone_backtest` and pass `ann` to `run_backtest_for_asset` (replace the literal `365.0`).
3. Add `--drop-weekends` flag to the `Backtest` subcommand in `main.rs` (`main.rs:~93+`).
4. Thread the flag through `main.rs:~223-245` config construction into `run_standalone_backtest`.
5. Default behavior (no flag): `drop_weekends = false` → `365.0` (unchanged for crypto). This preserves existing behavior.

## Validation
- `cargo test -p cdg` passes.
- Manual: `cdg backtest --coin btc --days 30 --drop-weekends` uses annualization factor `252.0`; without the flag it uses `365.0`.

## Why Included
Wrong Sharpe/annualized-return numbers in standalone backtest reports undermine alpha credibility. Fix is localized and low-risk.

## Note (out of scope)
`calculate_sharpe` in `backtest.rs` (hardcoded `365.0`, `code_review.md` #7) is intentionally out of scope for this phase (consistent with `alpha_readiness_plan` H4 which only fixes `run_standalone_backtest`). Tracked for beta; not alpha-blocking.
