# Phase 16: Add --light conflict warnings
**Priority:** P1 / base_plan P1  
**Module:** `src/pipeline.rs`, `src/main.rs`

## Goal
Make `--light` behavior explicit so alpha users understand what was overridden.

## Tasks
1. Add `warn_light_conflicts(config: &PipelineConfig) -> Vec<String>` in `pipeline.rs`.
2. Capture `original_days` before `config.days = 30` override. If `original_days != 30`, emit warning: `"--light overrides --days {n} -> 30"`.
3. If `config.coin.split(',').filter(|c| !c.is_empty()).count() > 1`, emit warning: `"--light expects a single coin; {n} supplied, only first used"`.
4. Fix misleading docstring in `main.rs:~52` (remove "forces coin=bitcoin" if present).
5. Print all warnings via `eprintln!` early in `run_pipeline_flow`, after the `--light` override block, before timestamp calculation.
6. Unit test the helper: warns for `(multi-coin)` and `(days=90)`; no warn for `(light=false)` and `(days=30)`.

## Validation
- `cargo test -p cdg` passes.
- Manual: `cdg collect --light --days 90 --coin bitcoin,ethereum` → two warnings printed.

## Why Included
Alpha users experiment with flags. Silent overrides cause confusion. Warning is cheap and improves perceived correctness.
