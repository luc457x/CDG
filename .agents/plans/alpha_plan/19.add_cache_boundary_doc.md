# Phase 19: Document cache boundary rounding behavior
**Priority:** P2 / base_plan P2  
**Module:** `doc/api_cache.md` (new), `src/coingecko.rs`, `src/pipeline.rs`

## Goal
Document the daily-boundary rounding behavior and TTL semantics so alpha users (and future contributors) understand why same-UTC-day runs share a cache key.

## Tasks
1. Create `doc/api_cache.md` documenting:
   - Cache key = full canonical URL (including query params).
   - Daily boundary rounding: `from`/`to` timestamps are rounded to UTC midnight (`(now / 86400) * 86400`), so runs within the same UTC day produce identical cache keys.
   - TTL distinction: URL pinning is separate from TTL expiration. A stale entry is evicted when `now - cached_at_timestamp > ttl_secs`.
   - Current TTL values per endpoint (market_chart, OHLC, tickers).
2. In `coingecko.rs`, add a single `eprintln!` or `tracing` line on cache **miss** so users understand why a fresh API call was made.
3. Update `README.md` cache section to reference `doc/api_cache.md`.

## Tasks (additions)
4. Add a unit test asserting two `now` values on the same UTC day produce identical `from`/`to` query params from `coingecko::get_coin_market_chart_range` (per `base_plan` P2).

## Validation (additions)
- Test: same-UTC-day `now` values → identical cache key params.

## Validation
- `cargo build` clean.
- Manual: run `cdg collect --coin btc` twice within one day → second run logs cache hit.

## Why Included
Alpha users will wonder why a fresh API call happens. A single doc + log line removes confusion without changing behavior.
