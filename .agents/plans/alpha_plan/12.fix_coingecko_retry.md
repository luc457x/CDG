# Phase 12: Fix CoinGecko retry logic for 5xx and network errors
**Priority:** P1 — Must fix before wider alpha distribution  
**Module:** `src/api/coingecko.rs` (alpha 2.4)

## Goal
Retry 429, 5xx, timeouts, and connection resets with exponential backoff. Currently only 429 is retried; everything else kills the pipeline.

## Tasks
1. Wrap `request_builder.send().await` in a `match` that inspects:
   - `reqwest::Error` → check if timeout or connect error → retry.
   - Response status → if `status.is_server_error()` (5xx) → retry.
   - 429 → retry with `Retry-After` honor (existing logic).
2. Use the same exponential backoff for all retryable errors.
3. Add a `reqwest::Client::builder().timeout(Duration::from_secs(30))` so infinite-hang requests fail fast.
4. Cap `max_attempts = 4` (or configurable constant) and return `Err` after exhaustion.

## Validation
- Unit test (using `wiremock`): mock 503 on first two requests, 200 on third → verifies retry succeeds.
- Unit test: mock timeout → verifies retry succeeds or fails cleanly after max attempts.
- `cargo test -p cdg` passes.

## Why Included
A single 503 from CoinGecko currently kills the entire backtest. Alpha users on shared/free APIs will see transient 5xx regularly.
